## 1. نظرة عامة على التحديثات

تم إجراء مراجعة هندسية شاملة لمنصة Madox لتحويلها من نظام بسيط إلى إطار عمل موثوق بمستوى الإنتاج. فيما يلي ملخص التحديثات الرئيسية:

| النظام | قبل التحديث | بعد التحديث |
|--------|------------|-------------|
| إدارة الجلسات | حفظ بسيط بدون نسخ احتياطي | 3 نسخ احتياطية متدورة + استرداد تلقائي |
| نظام السجلات | طباعة على الشاشة فقط | ملفات يومية + ضغط تلقائي |
| حالة البوت | تضيع عند كل إعادة تشغيل | محفوظة ومُستعادة تلقائياً |
| التحقق من الإعداد | لا يوجد | تحقق كامل قبل بدء التشغيل |
| معالجة الأخطاء | أخطاء صامتة | تتبع كامل مع بصمات الأعطال |
| مراقبة النظام | لا يوجد | مراقب صحة + تقارير تلقائية |
| الأمان | ثغرات متعددة | تحصين شامل |

## 2. الأنظمة الجديدة وكيفية عملها

### 2.1 مدير الجلسات (`utils/session.js`)

يتولى هذا النظام حفظ واسترداد ملف تعريف الارتباط (cookies) لحساب الفيسبوك.

**كيف يعمل:**
1. عند التشغيل، يحاول تحميل ملف `appstate.json` الرئيسي
2. إذا كان تالفاً أو مفقوداً، يبحث تلقائياً في 3 نسخ احتياطية:
   - `appstate.backup1.json` (الأحدث)
   - `appstate.backup2.json`
   - `appstate.backup3.json` (الأقدم)
3. عند كل حفظ ناجح، يدور النسخ الاحتياطية تلقائياً
4. يرفع الملف إلى GitHub مع إعادة المحاولة تلقائياً (حتى 3 مرات)

**المميزات:**
- ✅ كتابة آمنة (لا تلف عند انقطاع التيار)
- ✅ تحقق من صحة البيانات قبل الحفظ
- ✅ رفع إلى GitHub مع backoff تلقائي
- ✅ لا يتوقف البوت إذا فشل رفع GitHub

---

### 2.2 نظام السجلات (`utils/logger.js`)

يسجل هذا النظام كل أحداث البوت في ملفات منظمة.

**كيف يعمل:**
- ملفات السجل تُحفظ في مجلد `logs/`
- يومياً: ملف جديد باسم `madox-YYYY-MM-DD.log`
- إذا تجاوز الملف 5 ميغابايت: يُنشأ ملف جديد تلقائياً
- الملفات القديمة: تُضغط إلى `.gz` تلقائياً
- الملفات أقدم من 7 أيام: تُحذف تلقائياً

**مستويات السجل:**
| المستوى | الوصف |
|---------|-------|
| `DEBUG` | تفاصيل تقنية (يُظهر عند `LOG_DEBUG=true`) |
| `INFO`  | معلومات عامة عن التشغيل |
| `WARN`  | تحذيرات لا توقف البوت |
| `ERROR` | أخطاء تؤثر على الأداء |
| `FATAL` | أخطاء حرجة تمنع التشغيل |

**متغيرات البيئة:**
```
LOG_LEVEL=INFO    # الحد الأدنى للسجل (DEBUG/INFO/WARN/ERROR)
LOG_SILENT=true   # إيقاف الطباعة على الشاشة
LOG_DEBUG=true    # تفعيل وضع المطور
```

---

### 2.3 مدير حالة البوت (`state.js`)

يحفظ هذا النظام حالة البوت بشكل دائم بحيث تُستعاد بعد كل إعادة تشغيل.

**ما يتم حفظه:**
- المجموعات المقفولة (`lockedThreads`)
- المجموعات المكتومة مع وقت انتهاء الكتم (`mutedThreads`)
- معلومات المجموعات (`groupsCache`)
- إحصائيات كل مجموعة (`groupStats`)
- الردود التلقائية المجدولة (`autoReplies`)
- إعداد تأخير الرد (`replyDelay`)

**ملاحظة:** يحفظ تلقائياً كل دقيقتين وعند إيقاف البوت.

---

### 2.4 مراقب الصحة (`utils/health.js`)

يراقب هذا النظام صحة البوت وأداء الخادم باستمرار.

**ما يراقبه كل 30 ثانية:**
| المقياس | حد التحذير | حد الطوارئ |
|---------|-----------|------------|
| استهلاك الذاكرة (RAM) | 400 ميغابايت | 700 ميغابايت |
| تأخير حلقة الأحداث | 500 مللي ثانية | 3000 مللي ثانية |
| استهلاك المعالج (CPU) | 90% | — |

عند الوصول لحد الطوارئ: يُسجل الخطأ وينشئ لقطة تشخيصية تلقائياً.

---

### 2.5 محرك التشخيص (`utils/diagnostics.js`)

يتتبع الأخطاء وينشئ تقارير تفصيلية للمطورين.

**المميزات:**
- حفظ آخر 200 خطأ في الذاكرة مع stack trace كامل
- بصمة فريدة لكل نوع خطأ (تجنب التكرار في التقارير)
- تحذير عند تكرار نفس الخطأ 5 مرات
- إنشاء لقطة تشخيصية تلقائية عند تكراره 10 مرات
- ملفات اللقطات محفوظة في `data/snapshots/`

**الوصول للتشخيص:**
```
GET  /diagnostics           — تقرير الأخطاء والصحة
POST /diagnostics/snapshot  — إنشاء لقطة تشخيصية يدوياً
```

---

### 2.6 نظام الصيانة (`utils/maintenance.js`)

يُنظف النظام تلقائياً ويتحقق من صحته.

**عند كل تشغيل:**
1. إنشاء المجلدات الضرورية (`data/`, `logs/`, `backups/`, `data/snapshots/`)
2. حذف الملفات المؤقتة المتراكمة (صور uptime وغيرها)
3. التحقق من قابلية الكتابة في جميع المسارات الضرورية
4. التحقق من وجود التبعيات الأساسية

**يومياً الساعة 3 صباحاً:**
- تنظيف الملفات المؤقتة
- حذف السجلات القديمة (أكثر من 30 يوماً)

---

### 2.7 التحقق من الإعداد (`utils/config-validator.js`)

يفحص ملف `config.json` قبل تشغيل البوت ويمنع التشغيل بإعداد خاطئ.

**ما يفعله:**
- يتحقق من وجود جميع الحقول الإلزامية
- يتحقق من أنواع القيم (نص/رقم/مصفوفة)
- يطبق القيم الافتراضية للحقول الاختيارية المفقودة
- يُحذر من مفتاح API الضعيف
- يُحذر من إعدادات خارج النطاق الآمن
- يقرأ قيم حساسة من متغيرات البيئة (لا تُخزن في الكود)

## 3. الإعداد والتشغيل

### متطلبات التشغيل
```
Node.js >= 22.0.0
npm أو node مع صلاحيات كافية
```

### خطوات التشغيل الأولى
```bash
# 1. تثبيت التبعيات
npm install

# 2. وضع ملف الجلسة
# انسخ ملف appstate.json المصدَّر من متصفحك إلى جذر المشروع

# 3. تعيين متغيرات البيئة
export GITHUB_PERSONAL_ACCESS_TOKEN="رمز_الوصول_الخاص_بك"
export DASHBOARD_API_KEY="مفتاح_آمن_وعشوائي"

# 4. تشغيل البوت
npm start
```

### متغيرات البيئة المتاحة
| المتغير | الوصف | إلزامي؟ |
|--------|-------|---------|
| `GITHUB_PERSONAL_ACCESS_TOKEN` | رمز الوصول لحفظ الجلسة على GitHub | موصى به |
| `DASHBOARD_API_KEY` | مفتاح تأمين واجهة API | موصى به |
| `BOT_PREFIX` | بادئة الأوامر (يتجاوز config.json) | لا |
| `BOT_ADMIN_IDS` | معرّفات المديرين مفصولة بفاصلة | لا |
| `BOT_EMAIL` | بريد إلكتروني لإعادة تسجيل الدخول | لا |
| `BOT_PASSWORD` | كلمة مرور لإعادة تسجيل الدخول | لا |
| `LOG_LEVEL` | مستوى السجل (DEBUG/INFO/WARN/ERROR) | لا |
| `LOG_DEBUG` | تفعيل وضع المطور (`true`/`false`) | لا |
| `LOG_SILENT` | إيقاف الطباعة على الشاشة (`true`/`false`) | لا |
| `CORS_ORIGIN` | النطاق المسموح له بالوصول لـ API | لا |

## 4. شرح ملف الإعداد

```json
{
  "prefix": "-",                    // بادئة الأوامر (يمكن تغييرها)
  "appStatePath": "./appstate.json",  // مسار ملف الجلسة

  "bot": {
    "name": "Madox",               // اسم البوت
    "version": "2.0.0",            // رقم الإصدار
    "adminIDs": ["..."]            // معرّفات مديري البوت
  },

  "features": {
    "autoSaveAppState": true,      // حفظ الجلسة تلقائياً
    "autoSaveIntervalMs": 900000,  // كل 15 دقيقة (لا تضع أقل من 60000)
    "greetNewMembers": true,       // ترحيب بالأعضاء الجدد
    "farewellMembers": true,       // وداع الأعضاء المغادرين
    "antiSpam": true,              // الحماية من السبام
    "antiSpamCooldownMs": 3000     // فترة انتظار بين الأوامر (مللي ثانية)
  },

  "dashboard": {
    "apiKey": "ضع_مفتاحاً_قوياً_هنا", // أو استخدم DASHBOARD_API_KEY
    "port": 3001                   // منفذ واجهة API
  },

  "loginOptions": {
    "maxRequestsPerMinute": 40,    // الحد الأقصى للطلبات في الدقيقة
    "maxConcurrentRequests": 5,    // الطلبات المتزامنة
    "autoReconnect": true          // إعادة الاتصال تلقائياً
  }
}
```

## 5. تعليمات الصيانة

### الصيانة الدورية (تلقائية)
كل هذه العمليات تتم تلقائياً — لا تحتاج لتدخل يدوي:

| العملية | التوقيت |
|---------|---------|
| حذف ملفات temp | عند كل تشغيل + يومياً 3:00 ص |
| ضغط السجلات القديمة | كل 12 ساعة |
| حذف سجلات أقدم من 30 يوماً | يومياً 3:00 ص |
| حفظ الحالة | كل دقيقتين |
| حفظ الجلسة | كل 15 دقيقة (قابل للتعديل) |

### الصيانة اليدوية (عند الحاجة)

**مشاهدة السجلات الحية:**
```bash
tail -f logs/madox-$(date +%Y-%m-%d).log
```

**مشاهدة سجلات الأخطاء فقط:**
```bash
grep "\[ERROR\]\|\[FATAL\]" logs/madox-$(date +%Y-%m-%d).log
```

**إنشاء لقطة تشخيصية يدوياً:**
```bash
curl -X POST http://localhost:3001/diagnostics/snapshot \
  -H "Authorization: Bearer مفتاحك_هنا"
```

**مشاهدة تقرير الصحة:**
```bash
curl http://localhost:3001/health
curl http://localhost:3001/diagnostics
```

## 6. استكشاف الأخطاء وإصلاحها

### البوت لا يبدأ

**الخطأ:** `appstate.json contains placeholder data`
```
السبب: ملف الجلسة يحتوي على بيانات وهمية
الحل: صدّر cookies حساب الفيسبوك وأعد رفع الملف
```

**الخطأ:** `Missing required field: "bot.adminIDs"`
```
السبب: ملف config.json ناقص
الحل: تأكد أن config.json يحتوي على جميع الحقول المطلوبة
```

**الخطأ:** `Missing packages: @neoaz07/nkxfca`
```
السبب: التبعيات غير مثبتة
الحل: نفّذ: npm install
```

---

### البوت يتوقف بعد فترة

**الأعراض:** البوت يتوقف عن الرد بدون سبب واضح

**خطوات التشخيص:**
```bash
# 1. تحقق من السجلات الأخيرة
tail -50 logs/madox-$(date +%Y-%m-%d).log

# 2. ابحث عن أخطاء الذاكرة
grep "CRITICAL\|memory\|OOM" logs/madox-$(date +%Y-%m-%d).log

# 3. تحقق من لقطات التشخيص
ls data/snapshots/

# 4. استعرض تقرير الصحة عبر API
curl http://localhost:3001/diagnostics
```

---

### فشل رفع الجلسة إلى GitHub

**الأعراض:** `GitHub push failed` في السجلات

**الأسباب المحتملة:**
1. رمز الوصول منتهي أو غير صحيح
2. انقطاع الإنترنت
3. تجاوز حد طلبات GitHub API

**الحل:**
```bash
# تحقق من صحة الرمز
curl -H "Authorization: Bearer $GITHUB_PERSONAL_ACCESS_TOKEN" \
  https://api.github.com/user

# الجلسة تُحفظ محلياً بغض النظر عن GitHub
# البوت يعمل بشكل طبيعي حتى بدون رفع GitHub
```

---

### الأوامر لا تستجيب

```
1. تحقق أن البادئة صحيحة (افتراضياً: -)
2. تحقق أن المستخدم ليس في فترة anti-spam
3. تحقق من السجلات: grep "Command" logs/madox-*.log
4. جرب أمر -ping لاختبار الاستجابة
```

## 7. إجراءات الاسترداد

### استرداد الجلسة التالفة

النظام يفعل هذا تلقائياً، لكن إذا احتجت للتدخل اليدوي:

```bash
# 1. تحقق من النسخ الاحتياطية
ls -la appstate.backup*.json

# 2. استعادة يدوية من نسخة احتياطية
cp appstate.backup1.json appstate.json

# 3. أعد تشغيل البوت
```

### استعادة الحالة المفقودة

```bash
# الحالة محفوظة في
cat data/state.json

# إذا كان الملف تالفاً، احذفه ليبدأ البوت بحالة جديدة
rm data/state.json
```

### عمل factory reset كامل

```bash
# أوقف البوت أولاً
# احذف الحالة المحفوظة
rm -f data/state.json
# احذف جميع السجلات
rm -f logs/*.log logs/*.gz
# أعد تشغيل البوت
npm start
```

### استعادة البوت بعد عطل كامل

```bash
# 1. تحقق من لقطات التشخيص للفهم
ls data/snapshots/
cat data/snapshots/snap-LATEST.json | grep reason

# 2. راجع آخر سجل
tail -100 logs/$(ls logs/*.log | tail -1)

# 3. تحقق من ملف الجلسة
node -e "
  const d = JSON.parse(require('fs').readFileSync('appstate.json'));
  console.log('Cookies:', d.length, '| Valid:', Array.isArray(d) && d.length > 0 && !d[0]._README);
"

# 4. أعد تشغيل البوت
npm start
```

## 8. الأسئلة الشائعة

**س: هل تضيع إعدادات المجموعات (قفل/كتم) عند إعادة التشغيل؟**
ج: لا. جميع إعدادات المجموعات الآن محفوظة تلقائياً في `data/state.json`.

**س: ما معنى رسالة `Backup rotation complete`؟**
ج: تعني أن البوت أنشأ نسخة احتياطية جديدة من الجلسة ودوّر النسخ القديمة. هذا طبيعي جداً.

**س: هل يمكنني تغيير البادئة دون إيقاف البوت؟**
ج: لا. تغيير البادئة يتطلب إعادة تشغيل البوت بعد التعديل في `config.json` أو عبر متغير البيئة `BOT_PREFIX`.

**س: ماذا يعني `High event-loop lag`؟**
ج: يعني أن البوت مشغول جداً ويستغرق وقتاً طويلاً للمعالجة. يحدث عادةً مع كثرة الطلبات أو عمليات I/O بطيئة. إذا استمر، راجع لقطات التشخيص.

**س: هل GitHub ضروري لعمل البوت؟**
ج: لا. GitHub اختياري للنسخ الاحتياطي السحابي. البوت يعمل بشكل طبيعي بالجلسة المحلية فقط.

**س: كيف أعرف إذا كان البوت يعمل بشكل صحيح؟**
ج: نفّذ الأمر `-ping` في أي مجموعة، أو تحقق من:
```bash
curl http://localhost:3001/health
```
