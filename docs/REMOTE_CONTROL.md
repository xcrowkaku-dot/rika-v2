# دليل نظام التحكم عن بُعد — Madox Bot

## مقدمة

نظام التحكم عن بُعد يتيح لمشرف البوت إدارة أي مجموعة دون الحاجة للتواجد فيها.
يعتمد النظام على محرك اكتشاف المجموعات `ThreadScanner` المصمم للموثوقية والأداء العالي.

---

## لماذا كانت المجموعات تختفي سابقاً؟

### المشكلة الجذرية

النظام القديم كان يعتمد على **التخزين المؤقت السلبي** فقط:

```
رسالة واردة → تُخزَّن في groupsCache → تظهر في القائمة
```

هذا يعني:
- **المجموعات الصامتة** (لم يُرسَل فيها شيء منذ تشغيل البوت) = **غير مرئية**
- **إعادة تشغيل البوت** = فقدان كل المجموعات غير المحفوظة
- **أسماء قديمة** = لا يوجد تحديث تلقائي
- **أعداد الأعضاء** = مأخوذة فقط من `participantIDs` في الرسائل الواردة

### الأسباب التفصيلية

| السبب | التأثير |
|-------|---------|
| Cache سلبية فقط | المجموعات الجديدة لا تظهر إلا بعد استقبال رسالة |
| لا pagination في جلب القوائم | المجموعات بعد الحد الأول لا تُجلب |
| لا enrichment للبيانات | الأسماء ناقصة / قديمة |
| لا تحقق من صحة البيانات | Thread IDs فاسدة تُعيق العمل |
| لا fault isolation | خطأ في مجموعة واحدة يُوقف العرض كله |

## كيف يعمل نظام الاكتشاف الجديد؟

### المراحل الأربعة

```
[Phase 1] getThreadList مع pagination
     ↓
[Phase 2] دمج مع groupsCache الموجودة (استرداد المجموعات القديمة)
     ↓
[Phase 3] enrichment مُستهدف عبر getThreadInfo للإدخالات ذات الأسماء الناقصة
     ↓
[Phase 4] تحقق + رفع نسخة Cache + ترتيب وعرض
```

### Phase 1 — الجلب المُرقَّم

```
صفحة 1 (20 thread) → صفحة 2 (20 thread) → ... → حتى 400 مجموعة
```

- الحد الأقصى: 20 صفحة × 20 مجموعة = **400 مجموعة**
- تأخير 500ms بين الصفحات لتجنب Rate Limiting
- إذا فشلت صفحة → يُسجَّل الخطأ ويكمل من الصفحة التالية

### Phase 2 — الاسترداد من الذاكرة

المجموعات المخزنة في `groupsCache` (من رسائل سابقة) تُدمَج مع نتائج Phase 1.
هذا يضمن أن المجموعات التي لم تظهر في آخر الصفحات لا تُفقد.

### Phase 3 — إثراء البيانات

فقط المجموعات ذات الأسماء الناقصة أو القديمة (أكثر من 10 دقائق) تُرسَل إلى `getThreadInfo`.
الطلبات تُنفَّذ على دفعات من 4 مع تأخير 300ms للحفاظ على الاستقرار.

### Phase 4 — التحقق والعرض

- Thread IDs المشبوهة تُحذف (يجب أن تكون أرقاماً من 5-20 خانة)
- الأسماء الفارغة تُستبدل بـ threadID كبديل مؤقت
- القائمة تُرتَّب أبجدياً بالعربية
- Cache تُحدَّث بالبيانات الجديدة

## معاملة الأخطاء

### مبدأ العزل

كل مجموعة معالجة بشكل مستقل:

```
getThreadInfo(group_1) ✅ → تُضاف للقائمة
getThreadInfo(group_2) ❌ → تُسجَّل كخطأ، تُضاف بالبيانات الأساسية
getThreadInfo(group_3) ✅ → تُضاف للقائمة
```

خطأ في مجموعة واحدة **لا يوقف** باقي العملية.

### Fallback تلقائي

في حالة فشل الفحص الكامل، يعود النظام تلقائياً إلى `groupsCache` الموجودة
ويعرض ما لديه مع تنبيه المشرف.

## نظام التنقل في القائمة

```
-control remote
    ↓
[البوت يفحص المجموعات]
    ↓
1. مجموعة الشباب  (24 عضو)
2. مجموعة العمل   (8 أعضاء)
...
10. مجموعة العائلة (15 عضو)

تالي > | 0. إلغاء
    ↓
[المشرف يكتب رقم مجموعة]
    ↓
[قائمة الأوامر]
    ↓
[تنفيذ الأمر في المجموعة المستهدفة]
```

### أوامر التنقل

| الأمر | الوظيفة |
|-------|---------|
| رقم (1-N) | اختيار مجموعة |
| `>` أو `تالي` | الصفحة التالية |
| `<` أو `سابق` | الصفحة السابقة |
| `0` | إلغاء |
| `00` | إلغاء (من قائمة الأوامر) |

---

## أوامر التحكم عن بُعد

| الرقم | الأمر | الوصف |
|-------|-------|-------|
| 1 | قفل/فتح الرسائل | منع الأعضاء من الكتابة |
| 2 | كتم/رفع الكتم | منع استجابة البوت |
| 3 | إرسال رسالة | إرسال نص للمجموعة |
| 4 | تغيير الاسم | تعديل اسم المجموعة |
| 5 | قفل/رفع قفل الاسم | تثبيت/رفع تثبيت الاسم |
| 6 | إحصائيات | عرض عدد الرسائل والأوامر |
| 7 | معلومات الأعضاء | عدد الأعضاء والمشرفين |
| 8 | لوحة التحكم | إرسال اللوحة الكاملة للمجموعة |
| 9 | الرد التلقائي | تفعيل/تعطيل/إعداد الرد |
| 0 | رجوع | العودة لقائمة المجموعات |
| 00 | إلغاء | إنهاء جلسة التحكم |

## تشخيص المشاكل

### الأمر -control scanner

```
-control scanner
```

يعرض:
- هل الـ API جاهزة؟
- هل فحص جارٍ الآن؟
- عدد الفحوصات الكلي
- تاريخ آخر فحص
- عدد المجموعات في الذاكرة
- مدة آخر فحص
- عدد أخطاء آخر فحص

### مشكلة: المجموعة لا تظهر

1. أرسل رسالة في المجموعة لتسجيلها تلقائياً
2. استخدم `-control remote` مجدداً (يُجري فحصاً جديداً كل 30 ثانية)
3. افحص `scanner` للتأكد من عدم وجود أخطاء

### مشكلة: الفحص بطيء

طبيعي عند عدد كبير من المجموعات. كل 4 مجموعات تحتاج ~300ms.
50 مجموعة = ~4 ثوانٍ تقريباً.

### مشكلة: البوت لا يستجيب للأرقام

تأكد أنك:
- لا تكتب أمراً بادئة `-` (تعامَل كأمر جديد)
- الجلسة لم تنته (5 دقائق للرد)
- الرقم ضمن النطاق المعروض

## الإعدادات التقنية

| المتغير | القيمة | الوصف |
|---------|--------|-------|
| `FETCH_LIMIT` | 20 | مجموعات per page |
| `MAX_PAGES` | 20 | الحد الأقصى للصفحات |
| `PAGE_DELAY_MS` | 500ms | تأخير بين الصفحات |
| `ENRICH_BATCH` | 4 | طلبات متزامنة للـ enrichment |
| `ENRICH_DELAY_MS` | 300ms | تأخير بين دفعات الـ enrichment |
| `INFO_TIMEOUT_MS` | 9000ms | timeout لكل طلب getThreadInfo |
| `SCAN_COOLDOWN` | 30s | الحد الأدنى بين الفحوصات |
| `STALE_AGE_MS` | 10 دقائق | عمر البيانات قبل إعادة الجلب |
| `PAGE_SIZE` | 10 | مجموعات per page في القائمة |

---

## إرشادات الصيانة

1. **أعد تشغيل البوت** إذا توقف الفحص تماماً
2. **افحص الـ Logs** بحثاً عن `ThreadScanner` لتتبع المشاكل
3. **لا تحذف `data/state.json`** يدوياً — يحتوي على cache المجموعات
4. **زيادة `MAX_PAGES`** إذا كان لديك أكثر من 400 مجموعة
